# Adding a new rate-limited consumer

The cookie-cutter walkthrough. samba's compounding payoff is here:
new upstreams = ~30 LOC of binary + a small chart, no new
infrastructure.

> **Today's only consumer is GitHub** (via `pleme-io/tend`). This
> document captures the shape so future consumers (Akeyless, Slack,
> Cloudflare, OpenAI, etc.) are easy when the time comes.

## The full surface area

For each new upstream you add:

```text
1. Rust impl (~30 LOC)         in your binary repo
   ├─ struct FooClient { ... }
   └─ impl UpstreamApi for FooClient

2. Helm chart (~50 lines)      in pleme-io/helmworks
   ├─ Chart.yaml (depends on pleme-lib)
   ├─ values.yaml (queueWorker.* + jetstream.*)
   └─ templates/*.yaml (one-line includes of pleme-lib)

3. K8s manifest (~30 lines)    in pleme-io/k8s
   ├─ namespace.yaml
   ├─ release.yaml (HelmRelease)
   └─ kustomization.yaml
```

That's it. Stream + consumer + worker + metrics + alerts + NetworkPolicy
+ ServiceMonitor are all generated by pleme-lib.

## Step 1: implement `UpstreamApi`

```rust
// in src/operator/foo_throttle.rs (your binary's repo)

use samba::UpstreamApi;
use serde::{Deserialize, Serialize};

#[derive(Debug, Serialize, Deserialize)]
pub struct FooRequest {
    pub resource_id: String,
    pub etag: Option<String>,
}

#[derive(Debug, Serialize, Deserialize)]
pub struct FooResponse {
    pub data: FooData,
    /// FOO API returns RateLimit-Remaining in headers; we propagate.
    pub rate_limit_remaining: Option<u32>,
    /// FOO returns RateLimit-Limit too; samba uses it for dynamic rpm.
    pub rate_limit_total: Option<u32>,
    /// 304-equivalent for adaptive shrinkage / cost accounting.
    pub unchanged: bool,
}

pub struct FooApi {
    client: reqwest::Client,
    base_url: String,
    token: String,
}

#[async_trait::async_trait]
impl UpstreamApi for FooApi {
    const NAME: &'static str = "foo";

    /// Cold-start fallback. samba switches to observed value once
    /// the first response with rate_limit_total arrives.
    const INITIAL_BUDGET_PER_HOUR: u32 = 1000;  // FOO's documented default

    type Request = FooRequest;
    type Response = FooResponse;
    type Error = anyhow::Error;

    async fn dispatch(&self, req: FooRequest) -> anyhow::Result<FooResponse> {
        let url = format!("{}/api/v1/{}", self.base_url, req.resource_id);
        let mut http_req = self.client.get(&url).bearer_auth(&self.token);
        if let Some(etag) = &req.etag {
            http_req = http_req.header("If-None-Match", etag);
        }
        let resp = http_req.send().await?;
        let status = resp.status();
        let remaining = resp.headers().get("x-ratelimit-remaining")
            .and_then(|v| v.to_str().ok()).and_then(|s| s.parse().ok());
        let total = resp.headers().get("x-ratelimit-limit")
            .and_then(|v| v.to_str().ok()).and_then(|s| s.parse().ok());

        if status.as_u16() == 304 {
            return Ok(FooResponse {
                data: FooData::default(),
                rate_limit_remaining: remaining,
                rate_limit_total: total,
                unchanged: true,
            });
        }

        let data: FooData = resp.json().await?;
        Ok(FooResponse {
            data,
            rate_limit_remaining: remaining,
            rate_limit_total: total,
            unchanged: false,
        })
    }

    fn rate_limit_remaining(&self, r: &FooResponse) -> Option<u32> {
        r.rate_limit_remaining
    }

    fn rate_limit_total(&self, r: &FooResponse) -> Option<u32> {
        r.rate_limit_total
    }

    fn was_cached_response(&self, r: &FooResponse) -> bool {
        r.unchanged
    }
}
```

Wire a CLI subcommand:

```rust
// in src/main.rs
Commands::FooThrottle { config } => {
    let cfg = samba::Config::load(config.unwrap_or("/etc/pleme-worker/config.yaml"))?;
    let api = FooApi { /* ... */ };
    samba::JetStreamPullWorker::new(api, cfg)?.run().await?;
}
```

## Step 2: create the Helm chart

Copy `pleme-io/helmworks/charts/pleme-tend-throttle/` to
`pleme-io/helmworks/charts/pleme-foo-throttle/`. Edit:

```yaml
# Chart.yaml
name: pleme-foo-throttle
version: 0.1.0
dependencies:
  - name: pleme-lib
    version: "~0.12.0"
    repository: "file://../pleme-lib"
```

```yaml
# values.yaml — five fields are all that change vs tend-throttle:
queueWorker:
  enabled: true
  image:
    repository: ghcr.io/your-org/your-binary
    tag: amd64-<sha>
  subcommand: "foo-throttle"
  upstream:
    kind: foo                       # ← changed
    budgetPerHour: 1000              # ← FOO's cold-start estimate
    budgetPctMax: 10
  rateLimit:
    quotaPct: 0.01                   # ← 1% of FOO's reported quota
    pressureWarnPct: 50
    pressureCriticalPct: 25
    jitterPct: 0.30
    burst: 1
    honorEtag: true
  nats:
    serverUrl: "nats://nats-pleme-nats.nats.svc:4222"
    stream: "FOO_JOBS"               # ← per-upstream stream
    consumer: "foo-throttle"          # ← per-upstream durable
    resultSubject: "foo.jobs.results"
    failedSubject: "foo.jobs.failed"
  auth:
    existingSecret: foo-token         # ← per-upstream secret
    secretKey: token
    envName: FOO_TOKEN

jetstream:
  enabled: true
  serverUrl: "nats://nats-pleme-nats.nats.svc:4222"
  streams:
    FOO_JOBS:
      subjects: ["foo.jobs.>"]
      retention: workqueue
      storage: file
      replicas: 1
      maxMsgs: 10000
      maxBytes: "100MB"
      maxAge: "24h"
      duplicateWindow: "10m"
      discard: old
  consumers:
    FOO_JOBS:
      foo-throttle:
        type: pull
        ackPolicy: explicit
        ackWait: "60s"
        maxAckPending: 1
        maxDeliver: 5
        filterSubject: "foo.jobs.refresh.>"
        replayPolicy: instant
```

The `templates/` directory is identical to tend-throttle's — eight
one-line includes of pleme-lib templates. Copy verbatim.

## Step 4: deploy

```sh
mkdir -p k8s/clusters/<cluster>/infrastructure/foo-throttle
```

`namespace.yaml`:
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: foo-system
  labels:
    kubernetes.io/metadata.name: foo-system
    pod-security.kubernetes.io/enforce: restricted
```

`release.yaml`:
```yaml
apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: foo-throttle
  namespace: foo-system
spec:
  interval: 10m
  dependsOn:
    - name: nats
      namespace: nats
  chart:
    spec:
      chart: pleme-foo-throttle
      version: "0.1.x"
      sourceRef:
        kind: HelmRepository
        name: pleme-charts
        namespace: flux-system
  values:
    queueWorker:
      rateLimit:
        quotaPct: 0.01
    # everything else inherited from chart defaults
```

`kustomization.yaml`:
```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - namespace.yaml
  - release.yaml
```

Add to your cluster's `infrastructure/kustomization.yaml`.

## What you DON'T have to do

Already provided by pleme-lib templates:

- ✓ Deployment with strict-singleton replicaCount + Recreate strategy
- ✓ ConfigMap rendering YAML config from values
- ✓ Service exposing metrics + health
- ✓ ServiceMonitor scraping `:9090/metrics` at 30s
- ✓ NetworkPolicy (deny-all + DNS + scrape + upstream HTTPS + NATS)
- ✓ PrometheusRule with the four standard alerts
- ✓ JetStream init Job (creates stream + consumer idempotently)
- ✓ Standard `samba_*` metric names (alert rules already match)
- ✓ Adaptive pressure shrinkage (free, in samba)
- ✓ ETag honoring on conditional responses (free, in samba)
- ✓ Dynamic rate-limit observation (free, in samba)

## Standard chart values

```yaml
upstream:
  kind: <slug>
  budgetPerHour: <int>             # cold-start estimate
  budgetPctMax: <int>              # 0–100, absolute cap

rateLimit:
  quotaPct: <0.0–1.0>              # ★ THE knob
  pressureWarnPct: <int>           # default 50
  pressureCriticalPct: <int>       # default 25
  jitterPct: <0.0–1.0>             # default 0.30
  burst: <int>                     # default 1
  honorEtag: <bool>                # default true

nats:
  serverUrl: <nats://...>
  stream: <STREAM_NAME>
  consumer: <consumer-name>
  resultSubject: <prefix>.results
  failedSubject: <prefix>.failed
  fetchTimeout: <duration>         # default "5s"

auth:
  existingSecret: <secret-name>
  secretKey: token
  envName: <YOUR_TOKEN_VAR>
```

That's the entire UX surface for adding a consumer. ~5 lines that
matter; rest is copy-paste defaults.
